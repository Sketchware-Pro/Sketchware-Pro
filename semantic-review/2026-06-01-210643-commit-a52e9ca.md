# File management, build error panel, and Code Project as default

This commit adds three features to the Code Project IDE: a file management context menu (new/rename/delete via long-press), a build error panel that replaces the old toast-only failure reporting with a scrollable clickable list, and a FAB behavior change that makes Code Project the single-tap action while demoting block project to long-press.

**Watch for:** Rename does not update open tabs referencing the renamed file, leaving the editor writing to a stale path (high, confirmed). The error panel regex uses the raw file path from compiler output, which may not resolve correctly when the compiler emits project-relative paths (high, likely). The `long_press_hint` string is declared but never displayed anywhere, so users have no way to discover block project creation (medium, confirmed).

## High-level view

Renaming a file that has an open tab leaves `OpenFileTab.file` pointing at the old path, so subsequent saves silently recreate the old file rather than writing to the new name. The `OpenFileTab.file` field is `final` — there's no mechanism to update it in place.

The build error panel parses compiler output with the regex `(.*\.(?:java|kt|xml)):(\d+):`. The click handler constructs `new File(filePath)` from the captured group and checks `.exists()`. If a build tool emits a project-relative path, the file resolves against `/` (Android's working directory), not the project source, and the click silently does nothing.

The FAB change removes the type-selection dialog from single-tap and moves it to long-press. The string `code_project_long_press_hint` exists in `strings.xml` but is never referenced in any layout or Java code — it's dead code, meaning there's no affordance telling existing users that long-press exists.

## Rename leaves open tabs pointing at stale paths

`showRenameDialog` performs `file.renameTo(renamed)` and then calls `refreshFileExplorer()`, but never touches `openTabs`. The `OpenFileTab` class stores `file` as a `final File` field — there's no setter. If the user renames `Foo.java` to `Bar.java` while it's open in a tab, the tab title still shows `Foo.java`, and the next save writes content to the old path, effectively recreating the deleted file.

The fix requires either making `OpenFileTab.file` mutable or closing and reopening the tab with the new File reference. The delete handler already demonstrates the pattern of iterating tabs by path:

```java
// After successful renameTo:
for (OpenFileTab tab : openTabs) {
    if (tab.getFile().getAbsolutePath().equals(file.getAbsolutePath())) {
        // update tab's file reference to 'renamed'
    }
}
```

## Error panel regex and path resolution

The regex `(.*\.(?:java|kt|xml)):(\d+):` works correctly for standard javac/kotlinc output that emits absolute paths:

```
/data/user/0/pro.sketchware/files/projects/601/src/com/example/Main.java:12: error: ';' expected
```

If the build tool emits a relative path (e.g., `src/Main.java:12:`), `new File("src/Main.java").exists()` resolves against the app's working directory (`/`), not the project source. The click does nothing because the file won't be found there.

The click handler should fall back to resolving against `project.getSourcePath()`:

```java
File file = new File(filePath);
if (!file.exists()) {
    file = new File(project.getSourcePath(), filePath);
}
```

Note: `BuildErrorAdapter` is a standalone class with no reference to `project`, so the fallback resolution lives in the `OnErrorClickListener` lambda inside `setupErrorPanel()` — that's the right place, since it has access to the activity's `project` field.

## FAB discoverability gap

The previous behavior showed a dialog on every FAB tap, presenting both "Block Project" and "Code Project" options. The new behavior directly launches `CreateCodeProjectActivity`. For existing users who have muscle memory of tap-to-choose, block project creation becomes effectively hidden with no migration path.

The `code_project_long_press_hint` string exists in `strings.xml` but is never referenced anywhere in the codebase. It was likely intended to be shown as a `Snackbar` or tooltip but was never wired up.

Android's `View.setTooltipText()` (API 26+) would surface the hint in accessibility tools. A one-time `Snackbar` after upgrade would be more visible to sighted users.

## Input validation gaps in file/folder creation

Both `showNewFileDialog` and `showNewFolderDialog` only check `!name.isEmpty()`. They do not reject names containing path separators (`/`), which would create nested structures or fail silently. They do not check for collisions with existing files — `createNewFile()` returns `false` if the file exists, but the return value is ignored.

The `createNewFile()` call is wrapped in `try/catch(Exception ignored)` — if creation fails for any reason (permissions, invalid name, disk full), the user sees the dialog dismiss and the explorer refresh with no new file and no error message.

## Thread safety of file operations

All file operations run on the UI thread inside dialog callbacks. `deleteRecursive` on a directory with many files is the only potential concern — a directory with 100+ files could take 10-50ms, approaching the 16ms frame budget. For large imported projects this could produce a dropped frame, but for the expected use case (small student projects) it's not an issue worth threading over.
