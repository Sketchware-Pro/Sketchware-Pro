# Fix six semantic review findings in Code Project mode (v3)

This commit addresses six previously identified issues: rename breaking open tab state, error-click path resolution, dead tooltip string, missing filename validation, silent rename failures, and unchecked `deleteRecursive`. The approach is disciplined — each fix is scoped precisely to the reported gap, validation and error reporting are consistent across all three dialog handlers, and the rename tab-update logic covers both direct files and child paths inside renamed directories.

**Watch for:** After `renameTo()` succeeds, `file.isDirectory()` is called on the *old* `File` object whose path no longer exists on disk — on Android's Linux VFS, stat() on a non-existent path returns false, meaning child-tab updates for directory renames silently don't fire (confirmed). The error-click fallback resolves relative paths against `getSourcePath()`, which is correct for javac/kotlinc output but could double-nest if a tool emits paths relative to a subdirectory of source root (possible).

**Verdict**: NEEDS_CHANGES — the `file.isDirectory()` post-rename bug is a confirmed logic error that breaks directory rename for open child tabs.

## High-level view

The rename tab-update logic handles the single-file case correctly (tab path updated, subtitle refreshed, active-tab case covered) but has a confirmed bug for the directory case: `file.isDirectory()` queries the *old* path after it has already been moved, so the directory branch of the child-tab update never executes. The fix is trivial — capture `file.isDirectory()` before the rename, or check `renamed.isDirectory()` after.

The filename validation covers the critical path-traversal vectors (`/`, `\`, `\0`, `.`, `..`) and is invoked in all three dialog handlers before any filesystem operation. It doesn't cap name length at 255 bytes (ext4 NAME_MAX), which is a UX gap but not a security issue.

The error-click fallback's two-stage resolution (try absolute, then relative to source root) covers the two shapes compiler output takes, and the final `file.exists()` guard prevents any crash or data corruption for malformed paths.

## Rename tab-update and the `isDirectory()` post-move bug

The rename handler's tab-update loop iterates all open tabs after `renameTo()` succeeds. For a direct path match it calls `tab.setFile(renamed)`. For children of a renamed directory, it checks `file.isDirectory() && tabPath.startsWith(oldPath + "/")` and reconstructs the child path under the new parent. The path reconstruction logic is sound: `relativePart` captures everything from the old root onward (including the leading `/`), and `new File(renamed.getAbsolutePath() + relativePart)` produces the correct new absolute path.

The bug: `file` is a `java.io.File` pointing at the *old* path. After `file.renameTo(renamed)` succeeds, that old path no longer exists on disk. `File.isDirectory()` calls `stat()` on the path — stat on a non-existent path fails, and `isDirectory()` returns `false`. The `else if` branch is dead code for all directory renames. Child tabs retain their stale paths, and subsequent saves will write to the old (now non-existent) location, silently creating a new file at the stale path instead of saving under the renamed directory.

Fix: capture `boolean wasDirectory = file.isDirectory();` before the `renameTo()` call, and use `wasDirectory` in the loop condition.

The subtitle update handles the active-tab case correctly: if the renamed file's tab is the active tab, the `tab.setFile(renamed)` call in the loop fires first, so when the subtitle refresh reads `openTabs.get(activeTabIndex).getFile()` it gets the new name. If no tabs are open (`activeTabIndex == -1`), the bounds check prevents an NPE.

## Filename validation coverage

`isValidFileName` rejects path separators (`/`, `\`), null bytes, and `.`/`..`. The check is applied *before* `new File(parentDir, name)` in all three dialog handlers, so there's no bypass path. What it doesn't check: names longer than 255 bytes (ext4 NAME_MAX) and control characters (U+0001–U+001F). Neither creates a security issue on Android, but excessively long names would produce an `IOException` from `createNewFile()` that is caught and surfaced as the generic "Operation failed" toast — so the failure mode is safe, just not user-friendly.

## Error-click path resolution

The two-stage resolution handles absolute paths (first `new File(filePath).exists()` check) and relative-to-source-root paths (fallback `new File(project.getSourcePath(), filePath)`). The resolution relies on `BuildErrorAdapter`'s regex delivering just the path component without a line-number suffix. If `filePath` arrives malformed (e.g., `Main.java:5`), neither resolution will find an existing file, and the final `if (file.exists())` guard no-ops — no crash, no data corruption, just a non-functional click.
