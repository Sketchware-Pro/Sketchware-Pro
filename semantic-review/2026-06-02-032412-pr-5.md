# Hardening the Maven dependency resolver: validation, XXE, atomic cache, and sync lifecycle

This commit works through the PR #5 review of the Code Project Maven resolver. Coordinate validation is pushed down into the `DependencyDeclaration` constructor so that transitive coordinates parsed from remote POMs can't escape the cache path; `PomParser` is hardened against XXE and taught to expand same-POM `${...}` property placeholders; artifact caching becomes atomic via a `.part` temp file; resolved jar names are disambiguated by groupId; and the sync UI now shares an in-progress guard with builds, wraps `resolve()` in a try/catch, clears stale jars before each run, and reads `dependencies.txt` from the editable source tree with a legacy fallback.

**Watch for:** `clearResolvedDir` deletes the previously-resolved jars on the main thread *before* the new sync runs, so a failed or partial sync leaves the project with fewer (or zero) dependency jars than it had ‚Äî a transient network failure silently degrades a working compile classpath (likely, medium). The resolver now marks an artifact resolved after its POM downloads but before its jar/aar downloads, so a jar-download failure still poisons that artifact for the rest of the run even though the POM-level fix removed the earlier poisoning (possible, low). The built-in `version` / `project.version` properties are written into the property map *after* `<properties>` is parsed, so they override any same-named entries the POM declared itself (possible, low).

## High-level view

The `resolvedSet` change is cycle-safe. The key is added the instant the POM download returns and before any recursion into transitive deps, so a cycle A‚ÜíB‚ÜíA terminates when the second visit to A finds the key already present. Moving the `add` after the download buys retry-after-failure across separate top-level declarations, but only there: a jar/aar download failure (which happens after the key is set) still marks the artifact resolved, and a transitive failure inside one subtree still aborts its siblings because the recursion isn't individually guarded (the latter is pre-existing).

Placeholder expansion is bounded and degrades safely. `resolvePlaceholders` caps itself at five passes, leaves unresolved `${...}` in place so the caller's `contains("${")` check skips the dependency, and never loops on self-referential properties. The one behavioral wrinkle is that the synthetic `version`/`project.version` entries overwrite POM-declared properties of the same name.

Constructor validation is contained. The only two construction sites are `parse()`, which pre-validates and returns null, and `PomParser`, which wraps construction in a try/catch and skips bad coordinates ‚Äî so throwing from the constructor doesn't surprise any caller, and `parse()` still returns null rather than throwing.

The atomic-cache rework is correct and the existing `length() > 0` cache check still holds, because the final cache path now only ever receives a fully-downloaded file. A `.part` left by a crash is harmless: the next attempt truncates it. Two asymmetries remain ‚Äî orphaned `.part` files are never reaped, and the rename has no copy fallback.

The sync lifecycle is sound. `isSyncing` is cleared in `onComplete`, `onError`, and the defensive `catch`, each before the `isFinishing` early-return, and the early `return`s all precede the point where the flag is set ‚Äî so the build menu can't stay disabled forever. The shared `isBuilding || isSyncing` guard serializes the two operations, though only the build menu item is greyed out, not the Sync item.

`dependencies.txt` is now written to and read from the source tree consistently, with a legacy-path fallback for old projects, and the stray file at the `src/main` root doesn't reach the compiler or packager.

The XXE hardening is the standard remote-XML lockdown; its only behavioral side effect is that a POM carrying a DOCTYPE now fails to parse and that dependency is skipped.

## resolvedSet now gates on POM download, not first encounter

The dedup key is `groupId:artifactId`, and the only change is *when* it enters the set: previously on first encounter, now immediately after `downloader.download(dep, "pom")` returns. Cycle safety is preserved. Walking A‚ÜíB‚ÜíA: resolving A downloads A's POM and adds `gA:aA` before iterating A's transitive list; descending into B adds `gB:aB`; B's transitive list points back at A; the recursive call for A finds `gA:aA` already present and returns. The only window in which A's key is absent is during A's own POM download, and no recursion happens inside that download, so the cycle can't slip through. Single-threaded resolution means two siblings naming the same artifact can't race through that window either.

The payoff of the move ‚Äî "a failed first attempt does not poison a later valid declaration" ‚Äî is real but narrower than the key name suggests. The key is added between the POM download and the jar/aar download, so an artifact whose POM succeeds but whose jar fails is still marked resolved; a later declaration of the same artifact reached through a different path won't retry the jar. And because the transitive loop

```java
for (DependencyDeclaration transitiveDep : pomInfo.dependencies) {
    resolveRecursive(transitiveDep, outputDir, resolvedJars, listener, depth + 1);
}
```

isn't individually guarded, any throw inside a subtree propagates up to the top-level `resolve()` loop and aborts the remaining siblings ‚Äî so the retry benefit only materializes across the separate entries in the top-level `declarations` list, not within a single dependency's tree. The subtree-abort behavior is pre-existing and out of scope here, but it bounds how much the timing change actually buys.

## Same-POM property expansion

`resolvePlaceholders` can't loop indefinitely: it caps at five passes and breaks the moment a placeholder key is missing from the map, leaving the literal `${...}` in the string. Both the dependency loop and `parseDependencyManagement` then reject any value still containing `${`, so an unresolved property cleanly drops the dependency rather than producing a bogus coordinate. A self-referential property (`a ‚Üí ${a}`) expands to itself and exits after the pass cap; a growing one (`a ‚Üí x${a}`) is likewise bounded. It resolves one placeholder per pass and bails on the first unresolved one even if a later placeholder in the same string would resolve ‚Äî immaterial for version fields, which are the only consumer.

The wrinkle is ordering: `parseProperties` runs first, then the code unconditionally does `properties.put("project.version", projectVersion)` and `properties.put("version", projectVersion)`. A POM that declares its own `<version>` property inside `<properties>` has it overwritten by the project's own version. Real POMs rarely define a property literally named `version`, but when one does, the synthetic value wins. Guarding the puts with `containsKey` would respect the POM's own declaration.

## Atomic cache writes via .part

Promotion to the real cache path happens only after the write completes, which makes the `cachedFile.exists() && cachedFile.length() > 0` cache check trustworthy ‚Äî the cache path can no longer hold a truncated, non-zero-length body from an interrupted write, which was the whole point. A `.part` left behind by a crash is overwritten by the truncating `FileOutputStream` on the next attempt for that artifact, so it never produces a false cache hit; the check looks at the final path, not `.part`.

Two minor asymmetries. Orphaned `.part` files are only ever reclaimed by a later download of the same artifact; an artifact that's never retried leaves its temp file in the cache indefinitely. And the promotion throws if `renameTo` returns false, with no copy fallback ‚Äî unlike `DependencyResolver`'s output copy, which falls back to a direct copy. Since the temp and the destination are always in the same directory, the rename is reliable and the missing fallback is unlikely to bite, but the two code paths handle the same situation differently.

## Stale-jar clearing happens before the sync can succeed

`clearResolvedDir(resolvedDir)` runs synchronously on the main thread, before the worker thread is even started, and deletes every file in `libs/resolved`. The intent ‚Äî so a removed or downgraded dependency doesn't linger on the compile/dex classpath ‚Äî is right, but the timing makes it destructive: the old jars are gone before the new resolution has produced anything to replace them. If the subsequent sync fails entirely (network down, repository unreachable, every dependency errors), `onError` fires with nothing copied and the directory stays empty; if it partially fails, only the dependencies that resolved this run survive. Either way a previously-working project ends up with fewer jars than before a transient failure, and the next build compiles against a thinner classpath until the user manages a fully successful re-sync.

Resolving into a fresh temp directory and swapping it in only after a clean completion, or diffing the new resolution against the existing directory and removing only what's no longer needed, would keep the last good set intact when a sync fails.

## Sync lifecycle and the shared in-progress guard

`isSyncing` is set only after the three early `return`s (missing `dependencies.txt`, empty declaration list), so none of those bail-outs disables the build menu. Once set, it's cleared in every terminal path: `onComplete`, `onError`, and the defensive `catch` around `resolve()` each set `isSyncing = false` and re-enable the menu *before* the `isFinishing()/isDestroyed()` early-return, so an activity tearing down mid-sync still resets the flag. Because `resolve()` always notifies the listener (or throws into the catch) and the network timeouts are finite, there's no path that leaves the menu disabled forever.

The partial-success path in `resolve()` calls `onComplete` and then `onError`; both run, so the flag is cleared twice and `progress.dismiss()` is called twice ‚Äî idempotent, no harm. One rough edge: `setBuildMenuEnabled` only toggles the build item, so the Sync action itself stays enabled during a build or an in-flight sync. Tapping it is caught by the `isBuilding || isSyncing` guard and silently returns, so it's safe, but the user gets no feedback and the menu state is inconsistent with how the build item greys out.

## dependencies.txt relocated to the source tree

The template writes `dependencies.txt` to `getSourcePath()` (`app/src/main`), and the sync reads from `getSourcePath()` first, falling back to the legacy `getProjectMyscPath()` root for projects created before the change ‚Äî so the write location and the primary read location agree, and old projects keep working through the fallback. The file explorer is rooted at `getSourcePath()`, which is the reason for the move: the file is now visible and editable where the legacy root location wasn't. The stray `dependencies.txt` sitting at the `src/main` root is inert for builds ‚Äî resource compilation walks only `res/` and the compilers collect only `java/` ‚Äî so it isn't compiled, linked, or packaged into the APK.

## XXE hardening

Disabling DOCTYPE declarations, external general/parameter entities, XInclude, and entity-reference expansion (with `FEATURE_SECURE_PROCESSING` as the backstop) is the standard lockdown for XML arriving from a remote source, which POMs are. The behavioral consequence worth knowing: with `disallow-doctype-decl` on, a POM that carries a DOCTYPE now throws during parse and that dependency is skipped. Standard Maven POMs don't declare a DOCTYPE, so this is an acceptable trade.
