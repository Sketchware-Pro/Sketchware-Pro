# Code Project mode — full-source IDE path for Sketchware Pro

This change introduces a parallel project type ("Code Project") alongside the existing block-based visual builder. A user can now choose between the two at project creation time. Code projects get a drawer-based IDE screen (sora-editor + file tree) and a build pipeline that runs AAPT2, ECJ, D8, and APK signing entirely on-device. The template scaffolds a minimal Hello World Android app with manifest, layout, strings, colors, styles, and a single Activity.

**Watch for:** The build pipeline invokes `Runtime.exec("zip")` to inject DEX into the APK — this binary doesn't exist on most Android devices (confirmed). AAPT2 is re-extracted on every build with no caching (confirmed). The builder runs on a raw `new Thread()` with no lifecycle awareness, so a screen rotation or back-press during build leaks the thread and the Activity reference (confirmed).

## High-level view

The project model layer (`CodeProject`) piggybacks on the existing `lC` metadata store by adding a single discriminator field (`sc_project_type=code`). Code projects pass through the same listing/sorting logic as block projects, relying entirely on adapter-level routing to prevent them from landing in the block design screen — there is no guard in `DesignActivity` itself.

The build pipeline (`CodeProjectBuilder`) duplicates the AAPT2 extraction logic from `ProjectBuilder.maybeExtractAapt2()` without the caching guard (`hasFileChanged`), so the ~5 MB binary is rewritten to disk on every build. It shells out to a `zip` command to add DEX to the APK, which will fail silently or crash on stock Android ROMs that don't ship a `zip` binary.

The file explorer adapter builds the full directory tree eagerly in-memory and uses manual `notifyItemRangeInserted/Removed` to expand/collapse nodes. The collapse logic has an index-tracking pattern that could produce inconsistent RecyclerView state if the tree is mutated concurrently (e.g., a build generating `gen/` while the explorer is open).

The IDE activity (`CodeProjectActivity`) ties the sora-editor widget to the file system via read/write-on-switch semantics. Undo history resets on every file switch, so accidental edits are irreversible once the user navigates away.

<summary>Issues (7)</summary>

1. **`zip` binary assumption in APK assembly** (confirmed) — `Runtime.exec("zip")` is not available on most Android devices. Replace with a Java-based zip implementation (ZipOutputStream to copy entries from resources.apk and add classes.dex).
2. **AAPT2 re-extracted every build** (confirmed) — `extractAapt2()` unconditionally overwrites the binary. Port the `hasFileChanged` guard from `ProjectBuilder` or at minimum check if the file already exists and is executable.
3. **Build thread leaks Activity** (confirmed) — `new Thread(() -> { ... CodeProjectActivity.this ... })` captures the Activity. If the Activity is destroyed mid-build, the thread holds a stale reference and `runOnUiThread` posts to a dead context. Use an executor + lifecycle-aware callback, or at minimum check `isFinishing()` before posting.
4. **AAPT2 link error detection is fragile** (likely) — The check `output.contains("error")` will miss warnings-as-errors and pass through if the word "error" doesn't appear in the locale's AAPT2 output. A process that crashes (SIGSEGV) produces no stderr at all.
5. **No build-button guard against double-tap** (confirmed) — `buildProject()` doesn't disable the menu item, so rapid taps launch parallel builds against the same output directories, corrupting artifacts.
6. **Editor undo history lost on file switch** (confirmed) — `binding.editor.setText(content)` resets the undo stack. The user's undo/redo history for the previous file is discarded with no warning.
7. **`appName` injected unsanitized into XML** (confirmed) — `CodeProjectTemplate.generateStrings` interpolates `project.getAppName()` directly into an XML resource file. Characters like `<`, `&`, or `'` will produce malformed XML.

<summary>Details</summary>

## APK assembly assumes host `zip` binary

`buildApk()` runs `Runtime.getRuntime().exec(new String[]{"zip", "-j", unsignedApk, dexFile})`. Android's `/system/bin` does not include a `zip` utility on most devices. The existing `ProjectBuilder` uses Java-level zip manipulation to inject DEX into the APK. The call also discards the Process's exit code — `waitFor()` returns it, but no check is made. If the command fails, the unsigned APK will contain only resources and signing will "succeed" on a broken artifact that installs but immediately crashes.

The fix: open the resources APK as a `ZipInputStream`, copy all entries to a new `ZipOutputStream` at the unsigned path, then add `classes.dex` as a new entry.

## AAPT2 extraction without caching

Every invocation of `build()` calls `extractAapt2()`, which unconditionally copies the architecture-appropriate AAPT2 from assets to `context.getFilesDir()`. The existing `ProjectBuilder` has a `hasFileChanged` guard that compares file sizes before extracting. For iterative builds (the common case in an IDE), this adds several seconds of unnecessary I/O on each build.

## Build lifecycle and thread safety

The build runs on an anonymous `new Thread(...)` that captures `CodeProjectActivity.this`. Android's configuration-change lifecycle means the Activity can be destroyed and recreated while the thread is alive. The `runOnUiThread` calls then post to a detached window, causing silent failures or `WindowManager` exceptions on toast display.

The `buildProject()` method also doesn't disable the build button, so rapid taps can launch parallel builds against the same output directories, race on file writes in `bin/`, and produce corrupt APKs.

## Template XML injection without escaping

`CodeProjectTemplate.generateStrings()` interpolates the app name directly:

```java
"    <string name=\"app_name\">" + project.getAppName() + "</string>\n"
```

If the user enters an app name containing `&`, `<`, `>`, or `'` (all valid in an app display name), the generated `strings.xml` will be malformed XML and AAPT2 will fail at resource compilation with an unhelpful error. The fix is to XML-escape the value (`TextUtils.htmlEncode` or manual replacement of the five XML entities).

## File explorer concurrent mutation risk

`FileExplorerAdapter` builds the full tree at construction time and exposes a `refresh(File)` method that rebuilds from scratch. During a build, `CodeProjectBuilder` creates `bin/`, `gen/`, and writes files under the source tree (`gen/` contains R.java). If the user opens the drawer mid-build, they see a stale tree. If they expanded nodes before the build and call `refresh()` after, all expand state is lost.

`collapseNode` performs arithmetic on list indices (`position + 1 + count`) while iterating children. If `notifyDataSetChanged` from `refresh()` fires while an expand animation is in-flight, RecyclerView can throw `IndexOutOfBoundsException`.

## Project routing relies on adapter-level check only

`ProjectsAdapter` checks `CodeProject.isCodeProject(projectMap)` to decide whether to open `CodeProjectActivity` or the block design screen. There is no equivalent guard in `DesignActivity` itself — if a code project's metadata is opened directly (e.g., via a deep link or restored from recent tasks), it would enter the block-based editor and likely crash or show an empty state. A defensive check in `DesignActivity.onCreate()` that rejects code projects would close this gap.

## AAPT2 error detection relies on string matching

The builder checks `output.contains("error")` to decide whether AAPT2 linking failed. This is locale-sensitive and misses crashes (SIGSEGV produces no stderr). Since `BinaryExecutor` doesn't expose the process exit code, the builder has no reliable way to detect failure. For the compile step, an empty-output check passes even when the process is still running — `BinaryExecutor.execute()` reads stderr to EOF but never calls `waitFor()`, so subsequent build steps can race against a still-running AAPT2 process.
