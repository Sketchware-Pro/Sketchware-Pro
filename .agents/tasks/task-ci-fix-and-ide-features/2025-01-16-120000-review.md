# Code Project IDE mode with CI fix, autocomplete, and Kotlin support

This change introduces a full "Code Project" IDE mode into Sketchware Pro alongside the existing block-based system. It adds a code editor activity with file tabs, a drawer-based file explorer, a project build pipeline (AAPT2 -> Kotlin -> Java/ECJ -> D8 -> APK -> sign), autocomplete suggestions, Kotlin language support, and a project creation/template flow. A small CI fix adjusts the mock google-services.json write target. The project creation hooks into the existing `ProjectsFragment` FAB and `ProjectsAdapter` tap handler.

**Watch for:** Kotlin compilation errors are silently swallowed — the exception says "Kotlin compilation failed" without including `collector.getDiagnostics()` (confirmed). The build thread has no lifecycle tie, so it continues executing after activity destruction (confirmed). The `SimpleCompletionItem` three-arg constructor may not match the pinned sora-editor version (likely). Closing a modified tab discards unsaved changes without prompting (confirmed).

## High-level view

The CI fix writes mock `google-services.json` to `$projectDir/google-services.json`, making it visible to all build variants. The old file at `app/src/debug/google-services.json` is still tracked in git and should be removed.

The file tab system holds content strings per open file in memory. The tab close logic handles index bookkeeping but never prompts for unsaved changes — user edits disappear silently on close.

The build pipeline mirrors the existing `KotlinCompiler` + `ProjectBuilder` patterns but operates on raw user-managed source files. Error reporting from the Kotlin step is incomplete: the `DiagnosticCollector` captures error messages but the thrown exception discards them, unlike the existing `KotlinCompiler.kt` which includes diagnostics in the exception.

The autocomplete is a hardcoded list of ~50 class names and ~50 method names matched by prefix. It stacks on top of sora-editor's own completion, making suggestions noisy — method names like `onCreate` appear in all contexts regardless of position.

## Build pipeline lifecycle and error reporting

The build runs on a bare `new Thread()` with no cancellation mechanism. `CodeProjectActivity` stores `isBuilding` as a volatile boolean but never uses it to cancel mid-build. When the activity is destroyed (back press during build, rotation, system kill), the thread continues executing and calls `runOnUiThread(() -> ...)` on a destroyed activity. The `isFinishing()` guard prevents crashes from Toast/subtitle updates, but the builder still consumes CPU/disk for a build nobody will see.

The Kotlin compilation step captures diagnostics but the thrown exception discards them:

```java
if (exitCode != ExitCode.OK) {
    throw new Exception("Kotlin compilation failed");
}
```

The existing `KotlinCompiler.kt` does `throw Exception(collector.getDiagnostics(areWarningsEnabled()))`, which surfaces actual error lines. Without this, users see "Build failed: Kotlin compilation failed" with no indication of which file or line caused the error.

There's also an ordering subtlety: the code checks `exitCode != ExitCode.OK` *after* deleting `META-INF`. If the exit code indicates partial failure, partial Kotlin output remains in `classesDir`, and the subsequent `compileJava()` step will see those partial class files on its classpath. This matches the existing `KotlinCompiler.kt` pattern but means a failed Kotlin step doesn't cleanly abort — it poisons the Java classpath.

## Tab close discards unsaved edits

When `onTabClose(position)` is called, the tab is removed from `openTabs` unconditionally. The `saveAllModifiedTabs()` method exists and is called before build and on back press, but the close-tab action (tapping the X) does not check `isModified()` or prompt the user. This is a data-loss path that users will hit frequently in normal editing workflows.

## AAPT2 error detection asymmetry

The AAPT2 compile step uses `if (!output.isEmpty()) throw ...` which treats any stderr content as fatal. `BinaryExecutor` reads only the error stream, and AAPT2 can emit warnings to stderr even on success. The AAPT2 link step in the same file uses a better strategy — checking whether the output file (`resources.apk`) exists. The compile step should adopt the same pattern or the builder should expose and check process exit codes.

## File explorer never refreshes

`FileExplorerAdapter` builds the directory tree eagerly in its constructor. The `refresh()` method exists but is never called from `CodeProjectActivity`. After creating a new file or after a build generates `R.java` in `genDir`, the explorer remains stale. Users would need to close and reopen the activity to see file system changes.
