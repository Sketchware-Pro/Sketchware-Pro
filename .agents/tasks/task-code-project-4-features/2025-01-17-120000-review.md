# Persistent undo, APK install, dependency JARs, and LogCat panel for code project mode

Four features are wired into the code project IDE: per-tab undo history using sora-editor's `Content` object sharing, a post-build dialog that installs the APK via FileProvider, a `libs/` folder convention that places JARs on both compile and dex classpaths, and a live logcat viewer backed by `Runtime.exec("logcat")`.

**Watch for:** LogcatPanel leaks a native process and reader thread if the activity is recreated without `onDestroy` running to completion (confirmed). The logcat `addLine` path posts one Runnable per line to the main handler, which under high-throughput logcat output can saturate the main thread with adapter notifications (likely). The `Content` sharing approach is sound for undo history but creates a subtle coupling where `setContent(String)` (called nowhere in this diff, but publicly exposed) silently destroys accumulated undo history (confirmed).

## High-level view

The undo-history feature replaces the string-buffer tab model with a shared sora-editor `Content` instance. The public `setContent(String)` method on `OpenFileTab` creates a new `Content` (discarding history) and could surprise future callers â€” no code in this diff invokes it, but it's a foot-gun.

The APK install flow fires `ACTION_VIEW` with a FileProvider URI after a confirmation dialog. There's no handling of `ActivityNotFoundException` if no installer is present on the device.

Library JAR support adds `getLibraryJars()` and calls it in three build steps (Kotlin compile, Java compile, D8 dexing). In `createDex()`, JARs are added as **program files** rather than library files, meaning D8 will attempt to dex all classes in the JAR â€” if a JAR bundles classes that overlap with `android.jar`, D8 will fail with duplicate-class errors and no clear diagnostic.

The LogcatPanel spawns a native `logcat` process and reads its stdout on a daemon thread, posting each line individually to the main thread. There's no batching of lines before notifying the adapter, so a burst of logcat output produces a burst of `notifyItemInserted` + `scrollToPosition` calls on the main thread.

## LogcatPanel process and threading lifecycle

`LogcatPanel.start()` spawns a system `logcat -v brief` process and reads its output line-by-line on a daemon thread. Each line is posted individually via `mainHandler.post(() -> addLine(logLine))`. Logcat on a busy device can produce hundreds of lines per second, particularly during app launch. Each `addLine` call triggers either `notifyItemInserted` + `scrollToPosition` (normal path) or `notifyDataSetChanged` (when trimming past `MAX_LINES`). This creates a per-line layout pass and scroll animation on the main thread.

A simple mitigation: accumulate lines in the reader thread into a local batch, then post once per frame (16ms) or at a fixed interval (100ms). The batch would be drained in a single `notifyItemRangeInserted` call.

The `stop()` method calls `process.destroy()` followed by `thread.interrupt()`. The null-assignment of `readerThread` after interrupt creates a small window where a concurrent `start()` call could race against a still-running stop. The `isRunning` volatile flag prevents double-start, but not start-during-stop â€” if `stop()` sets `isRunning = false` and a new `start()` fires before the old thread exits, the old thread's `readLine()` could still be posting lines momentarily.

## Content sharing for undo history

The remaining `setContent(String)` method creates a brand-new `Content(content)`, which carries a fresh empty `UndoManager`. Any call to `setContent` would silently nuke undo history. In this diff no code path calls it, but it's a foot-gun for future maintainers. Consider removing it or renaming to `resetContent` to signal the destructive semantics.

## Dependency JAR pipeline

In `createDex()`, JARs are added as **program files** (not library files). This means D8 will dex all classes in the JAR. If a JAR contains classes that conflict with `android.jar` (e.g., an annotation-only JAR that bundles `javax.annotation`), D8 will produce duplicate-class errors. There's no diagnostic message surfaced to the user if this happens â€” D8 throws a generic exception caught by the builder's outer try-catch, producing a "DEX creation failed" toast with an opaque message.

## APK install flow

The `startActivity(intent)` call in `installApk` is unguarded. On stock Android this is always safe (the system package installer handles the intent), but on stripped-down or enterprise devices, there may not be an activity that handles `application/vnd.android.package-archive`. This would throw `ActivityNotFoundException` and crash the app.
